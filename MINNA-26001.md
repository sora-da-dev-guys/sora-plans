## Requirements Summary

- Create a Sales Order List page at `/sale-orders` in `smartbackoffice`
- Display all sales orders in a paginated table with search, filter, sort, and tabs
- Reference full feature spec from `SALES_ORDER_LIST_PAGE.md` in the old backoffice repo
- Follow existing patterns from `CustomerListPage.tsx` for structure and style

---

## Implementation Notes

### Architecture Approach
Follow the exact same layered pattern as the Customer module:
`types → api function → hook → page component → router`

All existing code uses `src/apis/sbb-api/` (not `src/apis/smartbestbuys/` as AGENTS.md suggests — that seems to be a planned rename). Follow what's actually in the codebase.

### API Endpoint
- Main: `GET /sale-orders/table-data` — same `ITableData<T>` wrapper pattern
- Supporting: `GET /data/v1/user-data` — fetch all users for "Assigned To" filter dropdown

### Complexity Note — Phase It
The spec is large (7 tabs + 13+ filter options + search + sort + pagination + status indicators). Recommend phasing:

**Full spec in one go (confirmed by Master):**
- Table with all columns: Order Number, Customer Name, Status, Payment Status, Amount, Date, Due Date, Assigned To, Special Indicators
- Search: by order number, customer name, customer phone, customer email, customer PO number
- Filter panel: status (multi-select), payment status (multi-select), assigned user (dropdown), date range, advanced checkboxes (8 options)
- Tabs: 7 order category tabs
- Sort by amount (asc/desc/default)
- Special indicator icons: shortage, follow-up, cancellation, shipping
- Pagination: 10/25/50 per page
- Click row → `/sale-orders/:id`
- Create button → `/sale-orders/create` (placeholder route)

### Permission Check

### Key Design Decisions
- Use `useSearchParams` for state persistence (same as CustomerListPage)
- Debounce search inputs at 500ms (same pattern)
- Use `ITableData<ISaleOrderTableData_Data>` generic — don't reinvent the wrapper
- Page input "Go to page" — include same jump-to-page input as CustomerListPage

2. **Tabs State** — Each tab filters by a different combination of flags/statuses. Recommend managing active tab as a `searchParam` (e.g. `?tab=0`) for deep-linkability and refresh persistence.
