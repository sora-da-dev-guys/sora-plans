# Plan: User List Page

**ID:** SORA-26005
**Repo:** smartbestbuys-backoffice
**Date:** 2026-03-16

## Requirements Summary

- Implement a User List page at `/users` in **smartbestbuys-backoffice** (the existing React 18 app using Material-UI, Tailwind, CRA)
- The existing page (`src/pages/users/list/UserListPage.tsx`) uses a card-based layout with only a status filter — needs a full upgrade to a table-based layout matching the spec
- Spec requires: table view with ID, Full Name, Email, Role/Position, Status, Department, Last Login columns
- Spec requires: search by name/email, filter by department/role/status/activity, pagination, sorting
- Action buttons: View Profile, Create New User, Edit User, Reset Password, Activate/Deactivate

---

## Implementation Notes

### Current State Analysis

The **existing** `UserListPage.tsx` already:
- Uses `useUsers()` hook → calls `GET /users/v1/fetch` → returns `User_TableData[]`
- Has only a status (`enabled`) filter via `Filter` component
- Renders users as cards (via `UserCard.tsx`) — **no table**
- Has a "Create User" button gated by `PermissionEnum.HR_User_Write`

### What Needs to Change

The page needs a full rebuild from card-based → **table-based layout**, matching other list pages in the repo (e.g., `CustomerTableDataPage.tsx`).

**Key issues with current `User_TableData`:**
The current API type is missing: `role`, `department`, `lastLogin`. These fields are in the spec but **not returned by the current API** (`GET /users/v1/fetch`). Two options:
1. **Extend API** — update `smartbestbuys-api` to return these fields (requires backend change, out of scope for this plan)
2. **Use what's available** — render table with available fields now, add missing columns when API is ready

> **Decision:** Implement the table with available fields (`id`, `firstName + lastName`, `email`, `phoneNumber`, `username`, `enabled`). Add placeholder columns for `role`, `department`, `lastLogin` marked as `-` until API is extended. Document as follow-up task.

### Filter Strategy

The spec calls for filters: department, role, status, activity. Since `User_TableData` only has `enabled` (boolean) at the moment:
- **Status filter** → filter by `enabled: true/false` (already exists, keep it)
- **Search** → client-side filter on `firstName + lastName + email + username` (no pagination from backend, data is fetched all at once)
- Department / Role / Activity filters → UI stubs only (disabled/greyed), pending API extension

### Pagination

Current `useUsers()` fetches all users at once (no server-side pagination). Use **client-side pagination** (slice the filtered array). Page size: 10.

### Permission Gating

| Action | Permission |
|--------|-----------|
| Create User button | `HR_User_Write` |
| Edit User | `HR_User_Write` |
| Activate/Deactivate | `HR_User_Write` |
| Reset Password | `HR_User_Write` |
| View Profile | `HR_User_Read` |

Use existing `getUserPermissionInfo(PermissionEnum.HR_User_Write)` util and `ConditionTooltip` pattern from `CustomerTableDataPage`.

### Activate/Deactivate Action

Use existing `updateUserActivationByIdAPI` (`PUT /users/v1/{id}/update-status`). After success, re-fetch user list or update local state.

### Reset Password

Route to existing user profile page which has update-criteria flow. Or open a dialog — TBD based on existing `UserActionAlert` component.
